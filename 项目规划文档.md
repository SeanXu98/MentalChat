## 阶段一：环境搭建与验证

### 1.1 阶段目标

搭建完整的大模型微调开发环境，验证环境可用性，确保后续训练流程顺利进行。

### 1.2 具体任务

#### 任务1.2.1：硬件环境确认

**目标**：确认硬件配置满足训练需求

**检查项**：
- GPU型号确认（目标：NVIDIA RTX 4090，显存24GB）
- CUDA版本确认（建议：CUDA 12.x）
- 磁盘空间确认（建议：100GB以上可用空间，用于存储模型权重和训练数据）

**验证方法**：
- 通过系统命令查看GPU信息和显存大小
- 运行简单的GPU计算测试，确认CUDA正常工作

**注意事项**：
- 若显存不足24GB，需调整后续量化策略（如使用更激进的量化方案）
- 确保电源功率充足（RTX 4090功耗较高）
- 建议使用Linux系统（Ubuntu 20.04/22.04），Windows系统可能存在兼容性问题

#### 任务1.2.2：Python环境配置

**目标**：创建独立的Python虚拟环境，避免依赖冲突

**核心依赖清单**：

| 依赖库名 | 版本要求 | 用途说明 |
|---------|---------|---------|
| Python | 3.10+ | 运行环境 |
| PyTorch | 2.0+ | 深度学习框架 |
| Transformers | 4.36+ | 模型加载与训练 |
| PEFT | 0.7+ | 轻量化微调 |
| Accelerate | 0.25+ | 分布式训练支持 |
| Datasets | 2.15+ | 数据处理 |
| BitsAndBytes | 0.41+ | 4-bit量化 |
| Pandas | 2.0+ | 数据清洗 |
| FastAPI | 0.104+ | API服务 |
| Gradio | 4.0+ | 可视化界面 |

**验证方法**：
- 逐个导入核心库，确认无报错
- 检查PyTorch是否正确识别GPU

**注意事项**：
- PyTorch版本需与CUDA版本匹配，否则无法使用GPU
- 建议使用conda或venv创建独立环境，避免与系统Python冲突
- 记录所有安装的版本号，便于后续环境复现

#### 任务1.2.3：基座模型下载与验证

**目标**：下载Qwen2-7B-Instruct模型，验证模型可正常加载

**实现步骤**：
1. 配置Hugging Face镜像（国内网络环境需配置）
2. 下载模型权重文件（约14GB）
3. 验证模型加载和基础推理功能

**验证标准**：
- 模型权重文件完整下载
- 可成功加载模型和分词器
- 可进行简单的文本生成测试

**注意事项**：
- 模型文件较大，确保磁盘空间充足
- 下载过程中网络中断可能导致文件损坏，需验证文件完整性
- 建议将模型缓存到本地，避免重复下载

#### 任务1.2.4：QLoRA配置验证

**目标**：验证4-bit量化配置可正常工作

**验证标准**：
- 使用BitsAndBytesConfig加载4-bit量化模型
- 确认显存占用在合理范围内（量化后约4-6GB）
- 确认推理结果正常

**注意事项**：
- BitsAndBytes库在Windows上可能存在兼容性问题，建议使用Linux
- 若量化后显存仍过高，检查是否正确应用量化配置

## 阶段二：数据处理

### 2.1 阶段目标

将原始对话数据清洗、标准化、增强，最终输出符合训练要求的高质量数据集。

### 2.2 数据现状分析

**原始数据格式**（基于data.csv）：
- 字段：Conversation ID（对话ID）、Turn ID（轮次ID）、Input（来访者输入）、Output（客服回复）
- 特点：多轮对话结构，同一Conversation ID下有多个Turn

**数据量**：约8000组对话（需统计确认）

### 2.3 具体任务

#### 任务2.3.1：数据探查与统计

**目标**：全面了解数据特征，识别潜在问题

**探查内容**：
- 对话总数统计
- 对话轮次分布（单轮/多轮对话占比）
- 文本长度分布（Input和Output的字符数/词数）
- 识别异常数据（空值、超长文本、乱码等）

**输出产物**：
- 数据探查报告（包含统计图表）
- 异常数据清单

**注意事项**：
- 重点关注超长对话，可能超出模型上下文长度限制
- 识别包含敏感信息的对话，后续需特别处理

#### 任务2.3.2：数据清洗

**目标**：去除低质量数据，提升数据质量

**清洗规则**：

| 清洗类型 | 具体操作 | 处理方式 |
|---------|---------|---------|
| 去重 | 删除完全相同的对话对 | 直接删除 |
| 去重 | 同一问题相似回复（相似度>0.95） | 保留一条 |
| 空值处理 | Input或Output为空 | 直接删除 |
| 长度过滤 | Input<5字符或>500字符 | 直接删除 |
| 长度过滤 | Output<10字符或>1000字符 | 直接删除 |
| 乱码处理 | 包含非中文字符超过30% | 人工审核或删除 |
| 敏感词过滤 | 包含自伤、极端词汇 | 人工审核后处理 |

**清洗后预期**：保留约90%数据（7200条左右）

**注意事项**：
- 清洗前备份原始数据
- 每个清洗步骤记录删除的数据量和原因
- 敏感词过滤需结合心理咨询场景特点，避免误删有效数据

#### 任务2.3.3：数据纠错与规范化

**目标**：修正数据中的错误，统一表达规范

**纠错内容**：
- 错别字修正（可使用纠错工具辅助，人工复核）
- 标点符号规范化（统一使用中文标点）
- 繁简转换（统一为简体中文）
- 空格和换行符规范化

**规范化内容**：
- 统一客服回复的语气（温和、专业）
- 去除回复中的个人信息（姓名、电话等）
- 统一专业术语表达

**注意事项**：
- 纠错需保持原意，避免过度修改
- 专业术语的统一需参考心理咨询行业标准
- 建议抽样检查纠错效果

#### 任务2.3.4：数据格式标准化

**目标**：将数据转换为模型训练所需的标准格式

**格式设计**：

单轮对话格式（适用于单轮训练）：
```
{
  "instruction": "你是一名专业的心理咨询客服，请根据来访者的问题，给出专业、共情的回复。回复需包含：1.情绪回应 2.问题分析 3.专业建议 4.引导跟进",
  "input": "来访者的问题内容",
  "output": "客服的专业回复内容"
}
```

多轮对话格式（适用于多轮对话训练）：
```
{
  "conversation_id": "对话ID",
  "turns": [
    {"role": "user", "content": "来访者第一轮输入"},
    {"role": "assistant", "content": "客服第一轮回复"},
    {"role": "user", "content": "来访者第二轮输入"},
    {"role": "assistant", "content": "客服第二轮回复"}
  ]
}
```

**实现步骤**：
1. 根据Conversation ID分组数据
2. 按Turn ID排序，组装多轮对话结构
3. 设计并应用指令模板
4. 转换为JSON格式

**注意事项**：
- 指令模板的设计直接影响模型训练效果，需反复优化
- 多轮对话的上下文长度需控制在模型限制内（建议2048 token以内）
- 确保JSON格式正确，无语法错误

#### 任务2.3.5：数据增强

**目标**：扩充训练数据量，提升模型泛化能力

**增强策略**：

| 增强方法 | 适用场景 | 具体操作 |
|---------|---------|---------|
| 同义词替换 | 增加表达多样性 | 将关键词替换为同义词（如"焦虑"→"紧张""不安"） |
| 情绪强度微调 | 增加情绪表达层次 | 调整情绪描述的强度（如"有点难过"→"非常悲伤"） |
| 句式变换 | 增加句式多样性 | 改变句子结构但保持原意 |
| 回复扩展 | 丰富回复内容 | 在保持专业性的前提下，适当扩展回复内容 |

**增强比例**：将7200条增强至8000条左右

**注意事项**：
- 增强数据必须保持专业性，不能偏离心理咨询场景
- 避免过度增强导致数据质量下降
- 增强后的数据需抽样人工检查
- 建议只对训练集进行增强，验证集和测试集保持原样

#### 任务2.3.6：数据集划分

**目标**：将数据划分为训练集、验证集、测试集

**划分比例**：训练集80%（6400条）、验证集10%（800条）、测试集10%（800条）

**划分原则**：
- 随机划分，但确保各集合中对话主题分布均衡
- 同一Conversation ID的所有Turn必须在同一集合中，避免数据泄露
- 测试集数据完全不参与训练和参数调优

**输出产物**：
- train.json（训练集）
- valid.json（验证集）
- test.json（测试集）

**注意事项**：
- 使用固定随机种子，确保划分可复现
- 记录划分的随机种子和划分结果
- 验证各集合的对话长度分布是否均衡

#### 任务2.3.7：数据质量最终验证

**目标**：确保处理后的数据质量达标

**验证内容**：
- 格式正确性：所有数据符合JSON格式要求
- 字段完整性：instruction、input、output字段完整
- 长度分布：各字段的长度分布合理
- 内容准确性：随机抽样100条，人工检查数据质量

**验证标准**：
- 格式错误率<0.1%
- 抽样检查通过率>95%

**注意事项**：
- 发现问题需回溯修正，重新执行相关步骤
- 保存验证报告，作为项目文档的一部分

## 阶段三：模型微调训练

### 3.1 阶段目标

使用处理好的数据集，对Qwen2-7B-Instruct模型进行QLoRA微调训练，获得适配心理咨询客服场景的模型。

### 3.2 具体任务

#### 任务3.2.1：LoRA配置设计

**目标**：设计最优的LoRA微调参数配置

**核心参数设计**：

| 参数名 | 建议值 | 设计依据 |
|-------|-------|---------|
| r（秩） | 16 | 平衡效果与显存占用的最优值 |
| lora_alpha | 32 | 通常设为r的2倍 |
| target_modules | ["q_proj", "v_proj"] | 注意力层核心模块 |
| lora_dropout | 0.05 | 轻度正则化，防止过拟合 |
| bias | "none" | 不训练偏置参数 |
| task_type | CAUSAL_LM | 因果语言模型任务 |

**注意事项**：
- 参数设置需根据实际训练效果调整
- 记录所有参数设置及调整原因

#### 任务3.2.2：QLoRA量化配置设计

**目标**：设计4-bit量化配置

**核心参数设计**：

| 参数名 | 建议值 | 设计依据 |
|-------|-------|---------|
| load_in_4bit | True | 启用4-bit量化 |
| bnb_4bit_quant_type | "nf4" | 专为大模型设计的量化格式 |
| bnb_4bit_compute_dtype | "float16" | 计算精度 |
| bnb_4bit_use_double_quant | True | 启用双量化，进一步节省显存 |

**注意事项**：
- NF4格式比FP4精度损失更小
- 双量化可额外节省约0.5GB显存

#### 任务3.2.3：训练参数配置设计

**目标**：设计训练的超参数配置

**核心参数设计**：

| 参数名 | 建议值 | 设计依据 |
|-------|-------|---------|
| num_train_epochs | 3 | 小样本场景避免过拟合 |
| per_device_train_batch_size | 4 | 适配24G显存 |
| gradient_accumulation_steps | 4 | 等效batch_size=16 |
| learning_rate | 2e-4 | LoRA常用学习率 |
| weight_decay | 0.01 | 轻度正则化 |
| warmup_ratio | 0.1 | 预热学习率 |
| lr_scheduler_type | "cosine" | 余弦退火调度 |
| logging_steps | 10 | 训练日志频率 |
| save_steps | 500 | 模型保存频率 |
| eval_steps | 500 | 验证频率 |
| fp16 | True | 混合精度训练 |
| optim | "paged_adamw_8bit" | 优化器 |

**注意事项**：
- 学习率是影响训练效果的关键参数，可尝试[1e-4, 5e-4]范围
- batch_size和gradient_accumulation_steps的设置需根据显存实际情况调整
- 保存步数不宜过小，避免产生过多checkpoint文件

#### 任务3.2.4：训练流程执行

**目标**：执行完整的模型训练流程

**执行步骤**：
1. 加载4-bit量化的基座模型
2. 应用LoRA配置，创建PEFT模型
3. 加载训练数据和验证数据
4. 配置训练参数
5. 初始化Trainer
6. 启动训练

**训练监控指标**：
- Training Loss（训练损失）：应持续下降
- Validation Loss（验证损失）：关注是否过拟合
- Learning Rate（学习率）：观察调度曲线
- GPU Memory（显存占用）：确保稳定

**注意事项**：
- 训练前预估显存占用，确保不会OOM
- 训练过程中定期检查loss曲线
- 若验证loss开始上升，考虑提前停止训练
- 保存多个checkpoint，便于回滚

#### 任务3.2.5：训练过程调优

**目标**：根据训练情况调整参数，优化训练效果

**常见问题与解决方案**：

| 问题现象 | 可能原因 | 解决方案 |
|---------|---------|---------|
| 训练loss不下降 | 学习率过低 | 提高学习率至5e-4 |
| 训练loss震荡 | 学习率过高 | 降低学习率至1e-4 |
| 验证loss远高于训练loss | 过拟合 | 减少epoch、增加dropout、数据增强 |
| 显存溢出 | batch_size过大 | 减小batch_size、增加gradient_accumulation |
| 训练速度过慢 | 数据加载瓶颈 | 使用DataLoader加速 |

**注意事项**：
- 每次调整只改变一个参数，便于分析效果
- 记录每次实验的参数和结果
- 使用验证集评估，不要过度调参导致测试集泄露

#### 任务3.2.6：模型权重保存

**目标**：保存训练完成的LoRA权重

**保存内容**：
- LoRA权重文件（adapter_model.bin）
- LoRA配置文件（adapter_config.json）
- 训练参数记录

**注意事项**：
- LoRA权重文件通常只有几十MB，便于存储和传输
- 建议同时保存完整的checkpoint，包含优化器状态
- 保存时标注训练时间、参数配置等元信息

## 阶段四：模型评估与优化

### 4.1 阶段目标

对微调后的模型进行全面评估，确保达到项目目标，并根据评估结果进行优化迭代。

### 4.2 评估指标设计

#### 任务4.2.1：自动化评估指标

**目标**：使用可量化的指标评估模型效果

| 指标名称 | 评估内容 | 目标值 |
|---------|---------|-------|
| Perplexity | 语言流畅度 | <5.0 |
| BLEU | 与参考回复的相似度 | >0.3 |
| ROUGE-L | 关键信息覆盖度 | >0.4 |
| 回复长度 | 回复的字数分布 | 50-300字 |

**注意事项**：
- 这些指标仅供参考，不能完全代表模型质量
- 重点关注指标的变化趋势，而非绝对值

#### 任务4.2.2：人工评估维度

**目标**：从专业角度评估模型回复质量

| 评估维度 | 评估标准 | 权重 | 目标值 |
|---------|---------|-----|-------|
| 专业度 | 回复是否符合心理咨询行业规范 | 30% | ≥90% |
| 共情能力 | 是否体现对来访者情绪的理解和回应 | 30% | ≥90% |
| 格式规范性 | 是否包含"情绪回应+问题分析+专业建议+引导跟进" | 20% | ≥95% |
| 语言流畅性 | 表达是否流畅、无语法错误 | 10% | ≥95% |
| 安全性 | 是否避免极端建议、违规诊疗 | 10% | 100% |

**评估方法**：
- 从测试集随机抽取100条样本
- 邀请3名评估者（具备心理咨询背景更佳）独立评分
- 取平均分作为最终得分

**注意事项**：
- 评估者需具备心理咨询基础知识
- 使用盲评方式，避免评估者知道回复来源
- 建立评估标准培训流程，确保评估一致性

#### 任务4.2.3：对比评估

**目标**：与基座模型、其他方案对比，验证微调效果

| 对比对象 | 对比内容 | 预期结果 |
|---------|---------|---------|
| 微调前（基座模型） | 回复的专业度和共情能力 | 微调后显著提升 |
| 全量微调（若有条件） | 效果差异和成本对比 | LoRA效果接近，成本更低 |
| 其他基座模型 | 同数据集下的微调效果 | Qwen2效果最优 |

**注意事项**：
- 对比评估使用相同的测试集
- 确保对比的公平性

### 4.3 模型优化迭代

#### 任务4.3.1：问题诊断

**目标**：根据评估结果识别模型弱点

**常见问题及诊断方法**：

| 问题类型 | 诊断方法 | 可能原因 |
|---------|---------|---------|
| 专业度不足 | 分析低分样本，识别共性问题 | 数据专业性不够、训练不充分 |
| 共情能力弱 | 统计情绪回应的覆盖率 | 指令模板不够明确 |
| 格式不规范 | 统计格式合规率 | 数据格式不一致 |
| 回复过于简短 | 分析回复长度分布 | 训练过早停止、学习率过低 |
| 回复过于冗长 | 分析回复长度分布 | 训练过度、学习率过高 |

#### 任务4.3.2：针对性优化

**目标**：根据诊断结果进行针对性优化

| 问题类型 | 优化方案 |
|---------|---------|
| 专业度不足 | 补充专业数据、延长训练、调整学习率 |
| 共情能力弱 | 优化指令模板、增加情绪相关的数据增强 |
| 格式不规范 | 统一数据格式、强化格式约束的指令 |
| 回复长度问题 | 调整max_new_tokens、增加长度约束指令 |

**注意事项**：
- 每次优化后重新评估
- 记录优化过程和效果变化
- 避免过度优化导致其他指标下降

#### 任务4.3.3：最终验收

**目标**：确认模型达到项目目标

**验收标准**：
- 专业度≥90%
- 共情贴合度≥90%
- 格式合规率≥95%
- 安全性100%

**验收流程**：
1. 使用完整测试集进行评估
2. 人工评估+自动评估结合
3. 生成验收报告

**注意事项**：
- 验收测试集必须与训练数据完全隔离
- 验收报告作为项目文档的重要组成部分

**注意事项**：

## 附录：关键决策记录

### 为什么选择Qwen2-7B-Instruct？
- 中文适配性最优
- 参数量适中，平衡效果与成本
- Instruct版本自带指令跟随能力

### 为什么选择QLoRA而非LoRA？
- 显存占用更低（11G vs 20G）
- 适配单卡RTX 4090训练
- 精度损失可控（约2%）

### 为什么选择r=16？
- 实验验证的最优值
- r=8效果不足，r=32显存过高

### 为什么采用8:1:1的数据划分？
- 小样本场景的最优实践
- 兼顾训练样本量和评估有效性
